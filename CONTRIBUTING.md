# Contributing

Thank you for your interest in this repository.

This project is a small deterministic React + TypeScript demo maintained by a single independent author. Its purpose is narrow and specific:

**Notice -> Interpret -> Shape -> Govern**

The repository exists to demonstrate a bounded weak-signal routing architecture. It is not a general-purpose product, platform, or agent framework.

At this stage, open pull requests are not being accepted by default.

If you have a bug report, documentation correction, design question, collaboration inquiry, or licensing question, please contact:

**Stephen A. Putman**  
Email: **putmanmodel@pm.me**

## Scope

Please keep bug reports, questions, and suggestions inside the intended scope of the project.

This repository is a narrow bounded-routing demo. Its purpose is to show:

**Notice -> Interpret -> Shape -> Govern**

in a deterministic, bounded weak-signal architecture.

Useful feedback areas include:

- observer emissions
- interpretation scoring
- ambiguity handling
- behavior mediation clarity
- governor allow/deny behavior
- bounded-response design
- audit strip clarity
- deterministic demo behavior
- README/documentation mismatches
- build or reproducibility issues

## Before Reaching Out

Please keep in mind:

- this is a deliberately small demo
- the architecture is intentionally bounded
- deterministic behavior is a core design constraint
- suggestions that push the project toward autonomy, persistence, unbounded inference, or authority expansion may be outside scope

## Bug Reports

Bug reports are welcome by email.

Please include:

- short summary
- reproduction steps
- expected behavior
- actual behavior
- relevant scenario
- environment details if relevant

If screenshots or screen recordings help clarify the issue, include them when possible.

## Documentation Issues

If you notice README or documentation mismatches, unclear wording, setup issues, or inaccurate descriptions of the architecture, please send a note by email.

This kind of feedback is especially useful for a demo like this, where clarity and scope discipline matter.

## Design Discussion and Collaboration

You are welcome to reach out for:

- bug reports
- documentation issues
- design discussion
- collaboration inquiries
- licensing or commercial use

If you want to discuss the design, please keep the conversation grounded in the repository’s intended boundary:

- observers emit narrow findings
- interpretation remains provisional
- behavior mediation shapes bounded options
- governor decisions constrain consequence

## Suggested Subject Lines

To make routing easier, you may use subject lines like:

- `bounded-weak-signal-routing-demo Bug Report`
- `bounded-weak-signal-routing-demo Documentation Issue`
- `bounded-weak-signal-routing-demo Contribution Inquiry`

## Pull Requests

Open pull requests are not being accepted by default at this stage.

If you believe a change is especially important, please contact the author first before preparing a contribution. This helps keep the project aligned with its narrow purpose and current maintenance model.
